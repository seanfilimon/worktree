## Why SHA-1 Had to Go

Git was designed around SHA-1, a hash function from 1995. SHA-1 has two critical problems in 2025: it's cryptographically broken (practical collision attacks have been demonstrated since 2017), and it's slow by modern standards. Git's migration toward SHA-256 addresses the security concern, but SHA-256 is even slower than SHA-1 for large-scale hashing workloads.

W0rkTree chose **BLAKE3** — a cryptographic hash function released in 2020 that is built for speed without compromising security. BLAKE3 is derived from the BLAKE2 family and the Bao tree-hashing scheme, delivering several properties that are ideal for version control:

- **Parallelism by design.** BLAKE3 internally uses a Merkle tree structure, which means hashing can be split across all available CPU cores automatically. On an 8-core machine, BLAKE3 hashes data roughly 8x faster than SHA-256.
- **Single, fixed output function.** Unlike SHA-2, BLAKE3 serves as a hash, MAC, KDF, and PRF with a single algorithm, simplifying W0rkTree's cryptographic surface.
- **SIMD acceleration.** The reference Rust implementation uses AVX-512, AVX2, SSE4.1, and NEON intrinsics, saturating memory bandwidth on modern hardware. In our benchmarks, BLAKE3 hashes at over 6 GB/s on a single core of an AMD Zen 4 processor.

## Content-Addressable Storage Internals

Every object in W0rkTree — blobs, tree manifests, commits, dependency declarations — is stored in a **content-addressable store (CAS)**. The BLAKE3 hash of an object's serialized bytes becomes its address. If two objects have identical content, they share the same address and are stored exactly once.

This architecture has cascading benefits:

1. **Automatic deduplication.** When multiple trees reference the same file (a shared library, a configuration template, a vendored dependency), only one copy exists on disk. Git achieves some dedup through packfiles, but it requires periodic `git gc` runs and doesn't deduplicate across submodules. W0rkTree deduplicates at write time, unconditionally.

2. **Integrity verification is free.** To verify any object, W0rkTree re-hashes its bytes and compares the result to the address. Because BLAKE3 is so fast, full-repository integrity checks run in seconds, not minutes. We verify on every read by default — silent corruption is impossible.

3. **Immutable objects, simple concurrency.** Objects in the CAS are write-once. Multiple threads and processes can read and write to the store concurrently without locks on individual objects. A lightweight index maps addresses to storage locations, and new writes are atomic at the filesystem level.

## Storage Savings in Practice

We measured storage consumption on five real-world repositories after converting them from Git to W0rkTree:

- **Medium monorepo (12 GB working tree, 45k commits):** W0rkTree's on-disk store was **68% smaller** than the equivalent `.git` directory. The savings come from cross-tree deduplication, more efficient object encoding, and the elimination of redundant packfile delta chains.
- **Multi-team project with shared libraries:** Teams had vendored identical copies of internal packages across four subtrees. W0rkTree stored them once. Git stored four copies (one per submodule pack).
- **Asset-heavy repository (game project, 8 GB of textures):** Binary files that appeared in multiple branches were deduplicated automatically. Git LFS would have stored each version separately on the remote; W0rkTree's CAS collapsed identical versions into single objects, saving roughly **72%** on the server.

The combination of write-time dedup and efficient serialization means W0rkTree repositories typically consume **50–70% less disk space** than the equivalent Git repository — before any additional compression is applied.

## Chunked Hashing and Transfer

BLAKE3's internal Merkle tree structure opens up another optimization: **chunked verification during transfer.** When pulling objects from a remote, W0rkTree can verify each 1 KiB chunk independently as it arrives, rather than waiting for the entire object to download before checking integrity. This means:

- Corrupted transfers are detected and retried at the chunk level, not the object level.
- Resumable transfers are trivial — the receiver knows exactly which chunks it already has.
- Parallel downloads across multiple connections can be reassembled and verified without a final serial hashing pass.

This is the same principle behind Bao (BLAKE3's verified streaming cousin), and it makes W0rkTree's network protocol both faster and more resilient than Git's packfile-based transfer.

## The Bottom Line

BLAKE3 isn't just a faster hash — it's a foundational building block that enables W0rkTree's storage efficiency, integrity guarantees, and transfer performance. By choosing a hash function designed for modern hardware and parallelism, every layer of the system benefits: writes are faster, reads are verified, storage is smaller, and network transfers are resilient. It's one of those decisions where picking the right primitive early pays dividends across the entire architecture.
