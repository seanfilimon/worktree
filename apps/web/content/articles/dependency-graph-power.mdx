## The Missing Layer in Version Control

Traditional version control systems store files and track changes, but they have no understanding of how those files relate to each other. Git treats your repository as a flat collection of blobs — it has no concept of "this library is consumed by that service" or "changing this schema will break those three downstream consumers."

Developers have built elaborate tooling on top of Git to fill this gap: Bazel's BUILD files, Nx's project graph, Turborepo's pipeline configuration. But these are all afterthoughts — bolted onto a system that fundamentally doesn't understand project structure. W0rkTree takes a different approach by embedding the dependency graph directly into the version control layer itself.

## How Cross-Tree Dependencies Work

In W0rkTree, dependencies are declared at the tree level and resolved across the entire repository hierarchy. A dependency edge can point to a specific tree, a branch within that tree, or even a pinned commit:

```
# In services/api/.worktree/deps.toml
[dependencies]
libs/auth = { branch = "main" }
libs/database = { branch = "main" }
libs/shared-types = { commit = "a3f8c91" }
```

When you declare these relationships, W0rkTree builds a directed acyclic graph (DAG) that spans the entire nested tree structure. This graph is versioned alongside your code — it evolves with your project and is always consistent with the current state of the repository.

Dependencies can carry metadata: whether they are build-time or runtime, whether they are optional, and what interface contract they expect. This richness enables W0rkTree to make intelligent decisions about change propagation that no external tool could replicate.

## Automatic Impact Analysis and TODO Generation

The dependency graph's most immediately visible benefit is automatic impact analysis. When you modify a file in `libs/auth`, W0rkTree traverses the graph and identifies every tree that depends on the changed code. It then generates TODO items for each affected downstream consumer, assigned to the team that owns that tree.

This isn't a naive "something changed, rebuild everything" approach. W0rkTree analyzes the scope of the change and cross-references it with the declared interface boundaries. If the change is internal to the library — a refactor that doesn't alter the public API — downstream TODOs are informational. If the change modifies an exported interface, the TODOs are flagged as blocking and the dependent trees are marked as potentially incompatible.

This automatic surfacing of cross-team impact replaces the fragile "did you remember to tell the other team?" workflow that plagues large organizations. The dependency graph makes implicit coupling explicit and actionable.

## Intelligent Build Ordering for CI/CD

The dependency graph directly drives CI/CD pipeline construction. Rather than maintaining a separate pipeline configuration that mirrors your project structure (and inevitably drifts out of sync), W0rkTree generates optimal build orderings from the graph itself.

When a commit lands, W0rkTree performs a topological sort of the affected subgraph to determine the correct build and test order. Independent subtrees are parallelized automatically. If `libs/auth` and `libs/database` have no dependency relationship, their builds run concurrently. If `services/api` depends on both, its build waits for both to complete before starting.

This extends to deployment ordering as well. When combined with linked branches, the dependency graph ensures that deployments happen in the correct sequence — databases migrate before services start, shared libraries publish before consumers deploy.

## Why This Changes Monorepo Workflows

In a traditional monorepo, the dependency graph lives in the build system and the version control system is oblivious to it. This creates a fundamental disconnect: your build tool knows what depends on what, but your VCS treats every change as isolated. Pull requests don't surface cross-team impact. Code review doesn't account for downstream breakage. Merges happen without dependency validation.

W0rkTree collapses this gap. The dependency graph is a first-class citizen of the version control model, which means every operation — branching, merging, reviewing, deploying — is dependency-aware by default. Teams no longer need to manually coordinate across project boundaries because the system handles it automatically. The dependency graph transforms version control from a passive storage layer into an active development intelligence system that understands how your code fits together.
